# Verification System Design Review (interactive + AWS executor)

Audited at `origin/main` `f80508b`, 2026-07-03. Companion to `audits/verification-system-review-f80508b.yaml` (14 findings: 5 novel, 9 planned-insufficient; 9 candidates rejected with named compensating controls). Read-only: every change below is proposed, none executed.

## Verdict in one paragraph

This is a deliberately engineered, unusually self-aware verification system -- solidly above ordinary industry practice, not yet at its own stated frontier. The repo-wide layer (two-tier TAP-style CI, forward-fix RCA with deterministic gate-escape forensics, hermeticity-typed verifiers, terraform artifact-verdict binding) is frontier-grade in places. The per-change layer has one architectural soft spot the system's own INTENT doc diagnosed fourteen months of decisions ago and never fixed: the Verification Plan -- the only check that exercises the promised behaviour end-to-end -- is still executed and self-reported by the agent that made the change, in ephemeral chat. Maturity: **interactive = solid** (two high findings), **aws-executor = strong as designed** (one high, nothing built yet -- confirmed: no executor/Step Functions terraform exists on disk).

## Q1 -- Is it frontier?

**Strong, with three genuinely leading mechanisms.** (a) Gate-escape forensics: on any main failure, ci-rca computes a tri-state `vacuous_pass` datum and a deterministic `escape_mode` taxonomy (`check_ran_vacuously` / `check_absent` / `tier_misplaced` / `no_premerge_gate_by_design`) -- most industrial CI systems have nothing comparable (T1.13, shipped 2026-06-30). (b) The process definition itself is under CI: skills carry machine-readable behavioural invariants checked by `prompt_compliance`. (c) The terraform path has true provenance: the apply consumes the same `plan.bin` the guard inspected, `plan_sha` recorded, red-record refusal (Decisions 77/92).

Against the external checklist: TAP presubmit/postsubmit split **met**; hermeticity **partial** (verifier classes AST-gated post-T3.6, but VP commands are arbitrary shell and live-reader DQ stays on the gate until T1.6); mutation testing **absent** (T3.7 not started); differential fail-on-revert **partial** (`is_admitted` shipped; its only production caller simulates the revert); selection correctness **partial** (diff-aware test selection is real; verifier selection is dead -- below); flake detection **partial** (designed, unbuilt); provenance **partial** (terraform yes, code merge `strict=false` with post-merge re-proof, VP evidence ephemeral).

## Q2 -- Is the roadmap sufficient?

**Partial.** What is planned is architecturally right, and what shipped (T3.1 kernel/registry/same-PR guard, T3.6 hermeticity, T0.12.5 typed acceptance shape, T1.13 evidence bundle) is well built. Two structural problems:

1. **Sequencing.** The oracle-quality layer -- T3.7 mutation testing, CD.30's diff-coverage ratchet, Decision 66's Tier D semantic judge -- is explicitly *not* on the CD.17 executor-unfreeze critical path (trigger keyed to T4.2/T3.2/T3.3 only). Meanwhile the observed rec-acceptance oracles are almost all structural `grep -q` checks, which *pass* fail-on-revert while proving only text presence. Today a human `/plan -> /implement` pass re-derives real verification on top of each rec (observed: the plan for rec-2481 added behavioural tests its rec's grep lacked). Unfreezing removes that human control by the same event that makes the grep the merge oracle -- with nothing gating on its replacement (VF-11, high).
2. **Ownership holes.** Live gaps no item or rec covers: independent re-execution/persistence of the interactive VP (VF-01 -- INTENT Wave 3 designed `harness --plan`; the roadmap extraction never carried it and the INTENT doc is on the demote list); the verifier-coverage repair (VF-02/03); a deterministic V-tier floor in validate.py (VF-04 -- Decision 48:2414 called for exactly this rec; dedup search: never filed); an interactive-era graduation producer (VF-05).

Everything else is owned and needs building, not redesign: CD.29 (hard-gate flip; hermeticity precondition already satisfied), CD.38 (sound), T3.2 (on the unfreeze path, correctly), T2.15, T1.6. T4.9 -- the executor's entire verification transport -- is `deferred_post_mvp`, which is coherent only because T4.1's state machine ends at `file_pr` (CI + auto-merge close the loop in the MVP).

## Q3 -- Is verification comprehensive?

**Vacuous paths exist.** Per-check (limb A): the sampled V2 plans are honest and differential -- they invoke the changed feature, some against the live warehouse. Three surviving vacuity classes, all evidence-backed: (1) the one sampled V3 plan's post-deploy VP *commands* grep the author's own transcription in DECISIONS.md for a run URL instead of re-invoking the deployed system -- the live run is real, the machine-checkable verdict is circular; (2) structural rec oracles (above); (3) a production hard-gate verifier passing vacuously **today**: `AthenaViewsVerifier` reports "Views are fresh. Found 45 decisions" against the frozen pre-Decision-84 Iceberg copy, while main carries 74 decision headers and dec-114 exists (VF-03, observed live).

Set-completeness (limb B): process-covered, not machine-covered. The designed machine backstop cannot fire: `AthenaViewsVerifier` inherits `covers=["**"]`, so `check_coverage()` reports *every* path covered -- empirically, `check_coverage(['zzz/definitely_not_covered.xyz']) == []` -- and `--coverage` additionally exits 0 unconditionally and is wired into no workflow (VF-02, high). A changed file with no covering verifier merges with no signal, and the tool prints the opposite.

Rejected as defects, with property-matched controls named in the YAML: SKIPPED-on-creds-down (VP-level SKIPPED is outlawed; harness-level skips are re-proven by the 3-hourly Main Canary with real OIDC creds, failures filing ci-rca recs); advisory executor verification (recorded CD.29 baseline, flip sequenced behind now-complete hermeticity); `strict=false` merges (deliberate non-wedging, re-proven post-merge on the merged tree); the same-PR weaken case (guard live in `--pre`); the budget escape hatch (CI-forbidden, audited, observed unused).

## Q4 -- Script vs agent: the decisive recommendation

**Interactive: hybrid.** Keep the VP as author-written deterministic script steps -- and note the design already has a rare anti-circularity property worth preserving as an explicit invariant: *the oracle predates the code* (VP expected outcomes are authored at `/plan` and fresh-context critiqued before implementation exists, then executed in a separate `/implement` session). What's missing is not more judgment but independent execution: (a) XS -- persist the VP compliance table (with attempt counts) into the PR body; (b) M -- a registered `validate_vp_replay` check that, on PRs carrying their plan, re-executes the hermetic pre-deploy VP steps in pr-validate; (c) keep zero-context agents exactly where they are (plan-critique on the oracle, code-review on the diff) -- a fresh agent re-running shell commands adds cost, not independence, over CI; (d) V3 post-deploy: human-gated deploy stays, evidence becomes a durable PR artifact the code-review agent inspects, and re-invokable smokes graduate into the registry. This also hardens the roadmap ledger for free: `closes_criteria` flips currently trust the self-reported VP pass.

**AWS executor: script -- endorse CD.38 exactly as written** (GitHub Actions is the sole verifier runner; personas never produce verdicts; Step Functions waits on callbacks; >15-min determinism rides the ECS escape). Three contract amendments while it is still text: run the per-rec differential verify (fail-on-revert) as a *GitHub Actions job*, resolving the T3.1-c6 "executor verify-state" wording that currently contradicts CD.38's sole-runner rule (VF-12); add head SHA to the T4.9 correlation tuple with merge-time equality assertion -- the terraform no-TOCTOU discipline applied to code (VF-09); give every verdict-wait state a heartbeat/timeout routing to ci-rca (VF-10). Authorship axis: executor-era oracles are usually authored by a different agent than the implementer (structurally good); the risk is oracle *weakness*, fixed at admission (differential job), audit (mutation-on-target), and authoring time (file the Tier D rec Decision 66 promised).

## Q5 -- Questions not asked (highlights; full answers in YAML)

Who re-executes the VP? -- CI, for the hermetic subset; self-attestation fails property-matching for V3. Graduate VP steps? -- selectively, bar = live `is_admitted` + hermeticity; today graduation is structurally impossible (zero workflow references; VF-05) and the admission gate never actually executes against new entries (VF-06). Coverage gate or advisory? -- repair first (it cannot fire), advisory on PRs, gate V3 later per the INTENT's own progressive-enforcement design. Tier trust? -- honest today, machine-enforce the 12m floor (~50 lines) before tiers feed an unattended merger. Flakiness? -- no detection on either surface; add attempt accounting to the persisted VP table; re-run-to-green with an empty diff should auto-file a rec. Callback model? -- sound at this scale; the three failure modes above are contract-text fixes. Defence-in-depth overlap? -- complementary by doctrine (CD.29's per-rec vs repo-wide distinction is exactly right); CI-authoritative is the correct tiebreak. Same-PR guard? -- blocks weakening, admits new tautologies until VF-06 lands. Added: the exit-criteria ledger inherits the VP's trust model (fix via VF-01); human and executor PRs hit the same gate by design -- the seam is upstream vocabulary (T3.10 deferred), not the gate.

## Top five, in order

1. **VF-01** (high): CI re-execution + PR-body persistence of the VP -- one CI step inside the existing 5-minute budget; converts the system's central residual attestation into a machine fact.
2. **VF-02** (high): make `check_coverage` able to fire (explicit covers; treat `"**"` as non-covering; wire into pr-validate).
3. **VF-11** (high): put an oracle-quality precondition on CD.17 reversal / CD.29 ratification for unattended merges.
4. **VF-05** (medium): give the 4-day-old graduation registry an interactive producer, or T3.7 mutates an empty set.
5. **VF-09** (medium): one line in T4.9 -- bind verdicts to head SHAs before the contract is built against.
