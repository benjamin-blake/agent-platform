## Q2 -- The alternative, costed: `feasible-high-cost`

A manual/paper L1+L3 loop is *feasible today* because human operations bypass every missing code surface: the operator is the broker adapter, order lifecycle, and reconciliation loop, so L2-absent / L3-print-only / risk-sublayer-absent (`ROADMAP-PRODUCT.yaml:388-393`, `async_engine.py:292`, `main.py:129/:139`) does not block it, and the data plane underneath is production-grade. Pinned in the binding units -- the sole operator's attention:

- **One-time setup ~20-40 hours**: paper brokerage account, a daily signal-extraction path over the existing feature tables, a sizing rule (pivot v1's equal-weight top-N is arithmetic), pre-registered promotion criteria (the pivot's own non-negotiable discipline), a logging/reconciliation runbook.
- **Recurring ~2-5 hours/week**, market-hours-coupled and interrupt-driven (daily signal check, manual order entry, EOD log, weekly review), sustained **>= 13 weeks**.
- **Calendar >= one quarter** before promotion-grade evidence -- the pivot's own bar is "run it on paper for 3 months."
- **The hidden dependency (SEQ-04)**: an honest *alpha*-premise test needs a discovered-formula artifact. The PySR discovery substrate is `CD.3 state: pending` with T2.6 parked `deferred_post_mvp` behind a circular reactivation clause, and no validated formula artifact is evidenced in the repo. Without one, the loop exercises operations plumbing -- not the edge.

## Q3 -- Value vs delay: `sequencing-favored` (conditionally)

The scarce resource is operator attention -- the frozen executor consumes no capacity, so "platform delay" means diverted attention plus calendar slip, not compute. The interleave's claimed payoffs shrink under tracing:

1. **"Real trading telemetry feeding the T2.36/T3.x chain" is a category error (SEQ-03).** Decisions 95-97 define that chain as the *session-rooted agent trace/observation model* (telemetry_sessions as trace root; T2.36 relands the agent-telemetry write path; T3.20 captures agent turns). A manual paper loop emits none of it; its trading data would land in product tables (`strategy_runs`, `tca_events`, fills) that are themselves absent -- building those early *is* the resequencing decision, circularly.
2. **Alpha validation is real but slow and gated**: >= 3 months of daily-bar data for weak statistical power, and honest only once a formula artifact exists (CD.3-pending). Starting today buys little evidence sooner.
3. **TCA/execution learning under manual entry measures the operator, not the future system's execution policy**; residual operational learning is genuine but modest and deferrable.

Against that: 2-5 h/week of the most fragmentation-expensive attention class, diverted from a months-out platform MVP whose *near-term* critical path (T2.26 -> T2.36 -> T3.2 -> T3.3, per mvp-triage) is dual-use substrate the product's evaluation funnel needs later. Note the zero-telemetry blind spot (rec-2173, confirmed open) **strengthens sequencing**: the platform's improvement loop is iterating blind today, and only the in-frame telemetry tail cures that -- paper-trading data cannot. The verdict is conditional: renewed silently month-over-month while the alpha premise accrues zero evidence, it degrades toward too-close-to-call.
