# Plan

## Intent
Establish a manifest-driven, per-Lambda packaging-and-verification architecture as the platform's single
source of truth for Lambda config and bundling, and ratify it as governance so that future Lambda work
deploys + verifies per-Lambda instead of under a blanket freeze. Concretely: ship the enforcement
mechanism (finish T-1.7, ship T-1.8 + T-1.6), ratify candidate decisions CD.16 (per-Lambda deploy/verify
gating) and CD.24 (per-Lambda packaging manifests) as a numbered Decision, and NARROWLY reverse only
Decision 67's Lambda-deploy-freeze clause -- leaving its STRATEGIC-plan-freeze clause (CD.17 / T4.2) in
force. The fleet is expected to grow well beyond today's eight artifacts, so the deliverable is not a
one-time stub exercise but a self-enforcing invariant: every Lambda artifact MUST carry a schema-valid
manifest, and the V2/V3 verification tier of any change is computed from the manifest graph rather than
from where a file happens to live. T2.17 (DuckLake writer/reader) is the first downstream consumer.

## Plan Type
IMPLEMENTATION
(STRATEGIC remains suspended per Decision 67 / CD.17; the complexity heuristic is suspended per AGENTS.md
Temporary Operational Constraints, so this is authored as a single larger IMPLEMENTATION plan.)

## Verification Tier
V3 -- the ratification half performs a live `ops_decisions` warehouse write (DynamoDB id allocation ->
`OpsWriter` -> S3, with Athena read-back), exactly as PLAN-ducklake-cd31-ratification did. There is no
Lambda or Terraform deploy, so the V3 `[pre-deploy]`/`[post-deploy]` tags map to `[pre-write]`/`[post-write]`
against the ops warehouse. The mechanism half adds local V2-grade gates (manifest schema validation,
bundle-completeness `py_compile`, manifest-coverage, plan-scope gating, build file-list equivalence) that
run inside the same plan. The binding behavioural checks are: (a) the ratifying Decision lands as a real
`dec-NNNN` row in `ops_decisions_current` (not an outbox stub) and CD.16/CD.24 read `state: ratified`;
(b) every Lambda artifact's handler import-resolves against its manifest-staged bundle AND every declared runtime asset is present there. Per Step 12c: because this is a warehouse-write (not Lambda/Terraform-deploy) V3 -- mirroring PLAN-ducklake-cd31-ratification -- only the irreversible-write boundary steps carry `[pre-write]`/`[post-write]`; the static/unit/local-integ/gate rows are intentionally untagged (tagging a grep step `[pre-deploy]` would be meaningless).
