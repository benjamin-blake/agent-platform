# Plan

## Intent
Grant the self-hosted CI runner the DynamoDB write actions (`PutItem`, `UpdateItem`) it needs so the ci-rca agent can allocate rec IDs and file diagnoses directly to Athena. Without this, ci-rca correctly identifies CI failures but writes `pending-{uuid}` outbox entries that never surface in the planning queue -- defeating Decision 72 (RCA-as-Plan-Source).

## Bundled Recommendations
- rec-858 (High, S): Add dynamodb:PutItem/UpdateItem to runner IAM so ci-rca can file recs

Explicitly NOT bundled (deferred to follow-on plans):

- [ ] `terraform plan -target=...` (scoped) shows ONLY `aws_iam_policy.github_runner_ci` (and optionally its attachment) mutating; zero `aws_instance.github_runner` references in the diff.
- [ ] Post-apply, `aws iam get-policy-version` on the live policy default version contains BOTH `dynamodb:PutItem` AND `dynamodb:UpdateItem` (assertion-and, not assertion-or).
- [ ] The new policy `DefaultVersionId` is captured into a variable during the apply step and recorded in rec-858's closure resolution.
- [ ] A `workflow_dispatch` of `ci-rca.yml` against a known-failed run id completes without the portal's outbox-fallback warning. Specifically: `[PORTAL] DynamoDB unreachable or credentials missing` does NOT appear anywhere in the run output. This is the unambiguous failure-signature inversion -- absence proves the IAM fix took effect end-to-end on the runner.
- [ ] The same dispatch's `/tmp/ci-rca-output.json` artifact contains at least one quoted `"rec-[0-9]+"` token (the newly-filed rec id from `ops_data_portal --file-rec` returning successfully). Free-text mentions of pre-existing rec ids (rec-858, rec-859) in the agent's reasoning prose do NOT satisfy this -- the check targets the quoted JSON token specifically.
- [ ] rec-858 closed via `ops_data_portal --update-rec` with a resolution naming this branch AND the captured policy `DefaultVersionId`.
- [ ] Any duplicate ci-rca rec produced by the dispatch test (re-diagnosing the same PR #354 failure) is closed as `superseded` referencing rec-859 (the original manual filing).
- [ ] A new forward-gate recommendation is filed via `ops_data_portal --file-rec` capturing the constraint: "Pin AMI in aws_instance.github_runner before any future plan that includes the instance in scope." This survives plan archival and surfaces in the next /plan session.

## Context
- ci-rca run 26299337363 on branch `agent/ci-rca-harness-fix` correctly diagnosed PR #354's SLOC violation (`scripts/product_roadmap.py` at 631 SLOC > 500 limit) but wrote `pending-044bbd3b` to the runner's outbox instead of allocating a real rec id from Athena.
- Root cause located precisely: `scripts/sync_recommendations.py:67` calls `ddb.update_item(TableName="agent-platform-counters", ...)` to allocate the next rec id. The runner IAM policy at `terraform/ec2_runner.tf:135-144` (Sid `DynamoDB`) grants `GetItem`/`Query`/`Scan` but NOT `UpdateItem` or `PutItem`. The boto3 call returns `AccessDeniedException`, portal falls back to writing pending-UUID to outbox.
- `seed_counters()` (bootstrap path) uses `put_item`, hence including `PutItem` alongside `UpdateItem` for forward compatibility -- the runner will not run `seed_counters()` in normal operation, but having it scoped to the same `agent-platform-*` ARN means no future IAM round-trip if bootstrap ever runs from CI.
- PR #355 (Decision 74) fixed the EACCES + workflow_dispatch gap. The IAM gap is the last harness completion item.
- The dispatch test in VP step 5 will re-diagnose the same PR #354 failure that already produced rec-859 (manually filed). This will produce a near-duplicate rec. VP step 8 supersedes it to keep the queue clean -- this is expected and not a workaround.
- AMI drift: `data "aws_ami" "ubuntu_22_04"` with `most_recent = true` means any unscoped `terraform plan` may show `aws_instance.github_runner` requires replacement (newer Canonical AMI published). The `-target` flags in VP step 1 confine the apply graph to IAM resources only.
- Outbox is currently clean (no `pending-*` files) -- the prior pending entry was cleaned up during PR #355 work. No cleanup precondition required.
