# Plan

## Intent
Make T0.3 (personal AWS account + IAM Identity Center) immediately verifiable and T0.6 (Lambda-tooling-platform Terraform skeleton) immediately committable. The user has already created the personal AWS account (id `REDACTED-PERSONAL-ACCOUNT`) and enabled Bedrock model access (T0.4 substantially complete via the existing `personal-bedrock-profile` profile), but the IAM Identity Center permission sets and the platform Lambda Terraform surface do not yet exist. This plan authors the verification surface for any remaining T0.3 work, the runbook for completing it, and the `terraform validate`-passing skeleton for the six platform Lambdas under a new `agent_platform` AWS profile (ratifying `agent_platform` as the canonical platform profile name, superseding the `personal_ml` placeholder in INTENT v5). It does **not** run `terraform apply` (deferred to T2.1 per Decision 67 Lambda freeze). Feeds the North Star by unblocking the AWS-migration chain — once this plan ships, T0.3 verification + T0.6 Terraform skeleton + runbook are all in place, leaving only `terraform apply` against the personal account as the remaining gating action for T2.1.

## Context

**Why now:** The user has the personal AWS account and Bedrock access (account `REDACTED-PERSONAL-ACCOUNT`, profile `personal-bedrock-profile`). T0.3's account-creation exit criterion is met; the Identity Center + permission set criteria are likely not. T0.6 has no work done. This plan closes the gap so that the only remaining gating step before T2.1 is the user completing the Identity Center setup (via the runbook) and the team running `terraform apply` against the personal account once the Lambda-deploy freeze lifts.

**Why `agent_platform` (not `bblake_platform` or `personal_ml`):** Ratified by the planning-session human decision (Step 6 confirmation). Reasoning: `personal_ml` (INTENT v5 placeholder) embeds a personal/company distinction that does not survive end-state (post-migration there is only one AWS account for the platform); `bblake_platform` was the planning agent's initial suggestion but couples the profile name to a specific owner; `agent_platform` is generic enough to distinguish from any future agent-unrelated AWS profile (e.g. a billing-admin role) while specific enough to communicate intent. This plan ratifies the choice by updating the INTENT v5 placeholder.

**Why `terraform validate` is sufficient pre-merge:** `terraform validate` proves syntax + schema correctness against the AWS provider's resource definitions; it does not require AWS credentials and does not contact AWS. A `terraform plan` would catch additional issues (e.g. ARN references to resources that do not yet exist), but it requires AWS credentials and would either need a real-account read or a mocked provider — neither justified for a skeleton that does not deploy. T2.1's plan can add `terraform plan` as a deployment gate when it lifts the freeze.

**Why stub handlers, not Terraform-only zip files:** Per the user's Step 6 choice. Stub handlers (a) make the import path `src.lambdas.{name}.handler` real so T0.7a's implementation can drop in cleanly, (b) make the smoke test trivially writable (`test_lambda_stubs.py`), (c) communicate intent — a 501 response is a clearer signal than a 200 with no body when an unimplemented endpoint is hit during early integration testing.

**Why six stubs instead of one shared stub:** The six Lambdas have different downstream consumers and will diverge in T0.7a / T0.7b / T0.7c / T1.1 / T1.3 / T1.4. Authoring six separate stub files now mirrors the eventual handler topology and prevents a future "split the shared stub" refactor.

**Why the INTENT + ROADMAP amendments are in scope:** INTENT v5 has Open Question #2 (Part 9, line 630) explicitly unresolved: "Final SSO profile name + personal account id ... Recommendation `bblake_platform`. Confirmation needed: is the personal account the same as `personal-bedrock-profile`'s target (REDACTED-PERSONAL-ACCOUNT) or a new account?". The user's Step 6 confirmation provides both halves of that resolution: (a) the SSO profile name is `agent_platform` (overriding the v5 `bblake_platform` recommendation, decoupling the profile-name convention from the resource-prefix convention), and (b) the personal account REDACTED-PERSONAL-ACCOUNT is the migration destination (same as `personal-bedrock-profile`'s target). Resolving Open Question #2 is the legitimate purpose of follow-on planning sessions on a REPORT-ONLY INTENT — INTENTs document what needs confirmation; confirmations come during implementation planning. The INTENT amendment therefore (a) rewrites line 294's SSO profile decision section, (b) marks Open Question #2 as RESOLVED, (c) adds an `## Application Status` entry. ROADMAP-PLATFORM.yaml carries the `personal_ml` placeholder at three exit_criteria lines (T0.3 line 1359, **T0.5** line 1393, **T2.3** line 2164 — round-2 critique correction; an earlier draft mislabelled them as T0.4 and T2.4); the sweep aligns the YAML with the resolved INTENT and the Terraform/runbook/verifier. Without these edits, the YAML exit_criteria, Terraform, runbook, verifier, and INTENT would each disagree on the canonical profile name, breaking the agent-first single-source-of-truth principle (AGENTS.md).

**Why no `build_lambda.py` change:** The Terraform `data "archive_file"` block zips `src/lambdas/{name}/` directly at plan-time; this works for `terraform validate` and a future `terraform apply` without `build_lambda.py` involvement.
